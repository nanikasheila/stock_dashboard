# ADR-002: Source of Truth の宣言（JSON vs Neo4j）

| 項目 | 内容 |
|---|---|
| **ステータス** | 提案 |
| **日付** | 2026-02-25 |
| **決定者** | architect エージェント |
| **関連ファイル** | `src/data/history_store.py`, `src/data/graph_store.py` |

---

## コンテキスト

現在、同一エンティティ（履歴データ）が 2 つの永続化ストアに並行して書き込まれている。

| ストア | 実装 | 保存先 |
|---|---|---|
| `history_store` | JSON ファイル | `data/history/` |
| `graph_store` | Neo4j グラフDB | Neo4jインスタンス |

この設計には以下の問題がある。

### 問題 1: トランザクション管理の欠如

JSON と Neo4j の書き込みは独立したトランザクションで実行される。
JSON 書き込み成功 / Neo4j 書き込み失敗（またはその逆）のケースで、両ストアが不整合な状態になり得る。

```
# 現状の疑似コード（問題のある設計）
history_store.save(record)   # ← JSON に書き込む
graph_store.upsert(record)   # ← 失敗しても history_store は巻き戻らない
```

### 問題 2: どちらが正しいかが不明

コードを読んだだけでは、JSON と Neo4j のどちらが正規データ源なのかが判断できない。
整合性が崩れた際の修復手順も定義されていない。

### 問題 3: 可用性リスク

Neo4j がオフライン（ローカル開発・ CI 環境等）のとき、Neo4j に依存する読み取り経路が失敗する。
ただし `graph_store` には `_available` フラグによる縮退動作が実装されており、部分的に対処されている。

---

## 決定

**JSON ファイル（`data/history/`）を Single Source of Truth（唯一の正規データ源）と宣言する。**
Neo4j はセカンダリの検索インデックスとして位置づける。

### データフローの方向

```
書き込み:
  呼び出しコード
    ↓ write
  history_store（JSON） ← Source of Truth
    ↓ index（非同期または遅延）
  graph_store（Neo4j） ← 検索インデックス

読み取り:
  グラフ検索が必要な場合 → graph_store（Neo4j）
  それ以外              → history_store（JSON）
```

### 実装方針

1. `history_store` が `graph_store` を知る形（`history_store` → `graph_store` の依存）にする
2. `graph_store` への書き込みは `history_store.save()` の内部で行う（呼び出し元は意識しない）
3. `graph_store` への書き込みが失敗しても、`history_store` への書き込みはロールバックしない
4. 不整合が生じた場合は `graph_store` を JSON から再構築（リビルド）する手順を整備する

```python
class HistoryStore:
    def save(self, record: HistoryRecord) -> None:
        """Save record to JSON (SoT) and sync to Neo4j index.

        Why: Callers should not be aware of dual persistence.
             JSON write is the authoritative operation.
        How: JSON write first, then best-effort Neo4j sync.
             Neo4j failure is logged but does not raise.
        """
        self._write_json(record)           # SoT への書き込み（失敗時は例外を上げる）
        self._sync_to_graph(record)        # ベストエフォートでインデックス同期
```

## 根拠

### 可用性の確保

JSON ファイルは外部サービスへの依存がなく、Neo4j がオフラインでも全機能が動作する。
現状の `_available` フラグによる縮退動作と整合する。

### データフローの単方向性

JSON → Neo4j の一方向同期により、どのデータが正しいかが常に明確になる。
双方向同期や「どちらが新しいか」の判定ロジックが不要になる。

### 修復手順の明確化

不整合発生時は `history_store` の JSON を読み込み、`graph_store` を完全再構築する。
逆方向（Neo4j → JSON）の復元は行わない（JSON が SoT のため）。

### 既存設計との親和性

`data/portfolio/portfolio.csv` も同様に「CSV = SoT」として設計されており、
本決定はプロジェクト全体の SoT 思想と一致する（`docs/architecture/data-flow.md` 参照）。

---

## 結果

### 正の影響

- 整合性の判断基準が明確になる
- Neo4j オフライン時も全機能が動作する
- 書き込み経路が `history_store` に一本化され、呼び出し元コードが簡素化される
- 不整合時のリビルド手順が定義できる

### 負の影響

- `history_store` が `graph_store` に依存するようになる（依存方向の変化）
- Neo4j インデックスが最新でない状態が一時的に生じ得る（結果整合性）

### リスク

- **グラフ検索の遅延**: Neo4j への同期が遅延した場合、グラフ検索結果が古い可能性がある。グラフ検索は「古くてもよいデータ」（研究履歴の類似検索等）に限定し、リアルタイム性が必要な箇所には使用しない。
- **graph_store のグローバルミュータブル状態**: `embedding_client._available` フラグが `graph_store` の縮退動作に使用されている。このフラグは `st.session_state` へ移行する必要がある（ADR 候補: Streamlit グローバル状態の排除）。
