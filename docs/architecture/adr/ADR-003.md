# ADR-003: CQRS 分離による取引書き込みフロー

| 項目 | 内容 |
|---|---|
| **ステータス** | Accepted |
| **日付** | 2026-02-26 |
| **決定者** | architect エージェント |
| **関連ファイル** | `components/trade_form.py`, `components/trade_writer.py`, `components/data_loader.py`, `src/data/history_store.py`, `src/core/portfolio/portfolio_manager.py` |

---

## コンテキスト

ダッシュボードから取引を記録する機能を追加するにあたり、書き込みロジックの配置について判断が必要となった。

### 選択肢

1. **既存の `data_loader.py` に書き込みロジックを追加する**
   - 読み取り関数と書き込み関数が同一ファイルに混在する
   - `data_loader.py` は既に 2000 行超の大規模モジュール

2. **書き込み専用モジュールを分離する（CQRS パターン）**
   - 読み取り（Query）: `data_loader.py` が担う（現状維持）
   - 書き込み（Command）: 新規 `trade_writer.py` + `trade_form.py` が担う

### 判断が必要な理由

取引書き込みには以下の複合的な要件がある:

- JSON（Source of Truth）と CSV（ポートフォリオ）への二重書き込み
- JSON → CSV の順序制御（JSON 書き込み失敗時は CSV を更新しない）
- 複数 Streamlit セッションからの同時書き込みに対する CSV の排他制御

---

## 決定

**CQRS パターンで書き込みを分離する。**

```
components/trade_form.py（UI）
  └─→ components/trade_writer.py（ファサード）
        ├─→ src/data/history_store.save_trade()   (JSON)
        └─→ src/core/portfolio/portfolio_manager  (CSV)
```

---

## 理由

| 理由 | 説明 |
|---|---|
| **責務過多の回避** | `data_loader.py` は既に 2000 行超の読み取り専用モジュール。書き込み追加は責務過多となる。 |
| **書き込み順序の集約** | JSON first（Source of Truth）→ CSV の順序制御を `trade_writer.py` 一箇所に集約できる。 |
| **ファイルロックの局所化** | `filelock` による CSV 排他制御を書き込み側に限定し、読み取り側への影響をゼロにできる。 |
| **テスト容易性** | 書き込みロジックを Streamlit 非依存の `trade_writer.py` に置くことで、純粋な unittest が書ける。 |

---

## 影響

- `data_loader.py` は一切変更不要。
- 新規追加: `components/trade_form.py`（UI + バリデーション）
- 新規追加: `components/trade_writer.py`（書き込みファサード）
- `src/data/history_store.py` と `src/core/portfolio/portfolio_manager.py` には `save_trade`, `add_position`, `sell_position` の各メソッドが実装されていることが前提となる。
